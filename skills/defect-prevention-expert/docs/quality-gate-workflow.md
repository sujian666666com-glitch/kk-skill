# 质量门禁工作流规范

> **定位**：本规范定义「预防 → 评审 → 度量 → 改进」的全生命周期质量闭环，作为团队使用 `defect-prevention-expert` 进行质量保障的最佳实践参考。

---

## 1. 概述

质量门禁（Quality Gate）是一种在软件交付关键节点设置检查关卡的质量保障机制。本规范将其固化为「预防 → 评审 → 度量 → 改进」的标准工作流，确保缺陷在萌芽阶段即被发现和消除，并通过量化度量驱动质量持续进化。

### 1.1 核心价值

| 价值 | 说明 |
|------|------|
| 左移缺陷发现 | 在预防阶段即识别潜在风险，而非等到评审或测试阶段 |
| 降低修复成本 | 早期发现缺陷的修复成本远低于后期 |
| 统一质量标准 | 通过标准化流程，确保团队输出质量的一致性 |
| 量化质量水平 | 通过度量指标客观评估质量，而非凭感觉判断 |
| 持续改进 | 基于度量数据制定改进方案，实现质量螺旋上升 |
| 知识沉淀 | 每次评审和度量都是团队知识库的更新机会 |

## 3. 生成阶段

### 3.1 输入与输出

| 生成内容 | 输入 | 输出 | 推荐 Skill |
|---------|------|------|------------|
| 需求文档 | 业务背景、用户故事 | PRD / 需求规格说明书 | 通用对话生成 |
| 设计文档 | PRD / 需求文档 | 技术方案 / 架构设计 | `system-design` |
| **代码编写** | **设计文档 / 接口定义** | **可运行的业务代码（应用预防方法论）** | **通用编码能力 + defect-prevention-expert** |
| **测试用例编写** | **需求文档 / 代码逻辑** | **系统化测试场景（覆盖六类方法论）** | **test-generator + defect-prevention-expert** |
| 代码实现评审 | 设计文档 / 接口定义 | 可运行的业务代码 | 通用编码能力 |
| 测试用例评审 | 需求文档 / 代码逻辑 | 测试用例文档 | `test-generator` |

### 3.2 预防阶段质量要求

- **需求文档**：业务逻辑完整，边界场景明确，异常流程定义清晰
- **设计文档**：架构合理，数据一致性方案明确，扩展性考虑充分
- **代码编写**：**应用逆向操作、依赖踏空、并发冲突、新旧兼容等预防方法论指导编码，避免常见缺陷模式**
- **测试用例编写**：**系统化设计测试场景，覆盖六大方法论，确保测试覆盖完整性**
- **代码实现**：符合编码规范，逻辑正确，无明显的并发/安全漏洞
- **测试用例**：覆盖主流程和核心分支，边界条件充分

## 4. 评审阶段（defect-prevention-expert）

### 4.1 评审类型路由

根据生成内容的不同，路由到对应的评审阶段：

| 生成内容 | 评审阶段 | 分析重点 |
|---------|---------|----------|
| 需求文档（从零设计） | Stage 1: 需求设计 | 需求完整性、边界场景、异常流程 |
| 设计文档 | Stage 2: 研发设计 | 架构合理性、数据一致性、扩展性 |
| **代码编写** | **Stage 3: 代码编写** | **应用预防方法论指导编码，避免常见缺陷模式** |
| **测试用例编写** | **Stage 4: 测试用例编写** | **系统化设计测试场景，覆盖六大方法论** |
| 需求文档（已有 PRD） | Stage 5: 需求评审 | 需求漏洞、边界场景、异常流程 |
| 设计文档（已有方案） | Stage 6: 设计评审 | 架构合理性、数据一致性、扩展性 |
| 代码实现 | Stage 7: 编码实现评审 | 并发安全、依赖处理、版本兼容 |
| 测试用例 | Stage 8: 测试用例评审 | 补充遗漏的测试场景 |
| 代码提交前 | Stage 9: 代码评审 | 逻辑缺陷、安全漏洞、性能问题 |

### 4.2 六大方法论应用

评审时必须覆盖以下六大方法论，按阶段侧重应用：

| 方法论 | 需求评审 | 设计评审 | 代码评审 | 测试评审 |
|--------|---------|---------|---------|---------|
| 逆向操作 | ★★★ | ★★★ | ★★★ | ★★☆ |
| 依赖踏空 | ★★★ | ★★★ | ★★★ | ★★☆ |
| 并发冲突 | ★★☆ | ★★★ | ★★★ | ★★☆ |
| 新旧兼容 | ★★☆ | ★★★ | ★★★ | ★★☆ |
| 状态迁移 | ★★★ | ★★★ | ★★★ | ★★★ |
| 因果判定 | ★★★ | ★★★ | ★★★ | ★★★ |

### 4.3 评审输出

评审完成后必须输出：

1. **评审报告**：按 P0-P3 标注严重程度和优先级
2. **风险清单**：每个问题附改进建议
3. **待办事项**：明确责任人和截止时间

## 5. 度量阶段（defect-prevention-expert）

### 5.1 度量类型路由

| 度量场景 | 度量阶段 | 分析重点 |
|---------|---------|---------|
| 上线前 | Stage 10: 上线前质量检查 | 质量基线达标情况、风险可控性 |
| 上线后 | Stage 11: 运维监控与度量 | 质量趋势、改进机会、根因分析 |

### 5.2 度量指标体系

| 质量维度 | 度量指标 | 目标值 | 数据来源 |
|---------|---------|--------|---------|
| 可用性 | 系统可用性（SLA） | ≥ 99.9% | 监控系统 |
| 性能 | 平均响应时间 | ≤ 200ms | APM 工具 |
| 可靠性 | 错误率 | ≤ 0.1% | 日志系统 |
| 缺陷逃逸 | 线上缺陷数/缺陷密度 | 逐月下降 | 缺陷管理系统 |
| 用户满意度 | 投诉率、NPS | 逐月改善 | 用户反馈系统 |
| 代码质量 | 代码覆盖率 | ≥ 80% | CI/CD 流水线 |

### 5.3 度量输出

度量完成后必须输出：

1. **质量度量报告**：包含各维度指标的目标值、实际值、趋势
2. **问题与改进清单**：按优先级排序的改进建议
3. **改进方案**：短期/中期/长期改进计划，明确责任人和时间节点

## 6. 改进阶段

### 6.1 改进分类与转交

| 改进类型 | 转交 Skill / 团队 | 说明 |
|---------|----------------|------|
| 具体 Bug 修复 | `bug-fixing` | 零回归修复工作流 |
| 代码结构优化 | `refactoring` | 安全重构，保持外部行为不变 |
| 测试用例补充 | `test-generator` | 生成测试用例代码 |
| 架构图/流程图更新 | `diagram-generator` | 绘制或修订架构图/流程图 |
| 深度代码评审 | `code-review` | 系统性代码审查 |
| 产品质量改进 | 产品+研发团队 | 基于度量数据的系统性改进 |

### 6.2 改进后复评

改进完成后必须重新进入度量阶段，确保：

- [ ] 改进措施已实施
- [ ] 度量指标有改善趋势
- [ ] 未引入新的缺陷或问题
- [ ] 相关文档已同步更新
- [ ] 测试用例已补充并通过

### 6.3 改进效果跟踪

| 跟踪项 | 说明 |
|--------|------|
| 改进措施完成率 | 计划中的改进措施是否按时完成 |
| 度量指标改善率 | 目标指标是否达到预期改善效果 |
| 缺陷逃逸率变化 | 改进后缺陷逃逸率是否下降 |
| 用户满意度变化 | NPS 或投诉率是否改善 |

## 6. 最佳实践

### 6.1 各阶段时机

| 阶段 | 触发时机 |
|------|---------|
| **需求设计** | 业务需求提出后，技术方案确定前 |
| **研发设计** | 需求文档评审通过后，编码开始前 |
| **需求评审** | 需求文档初稿完成后立即进行 |
| **设计评审** | 技术方案确定后、编码开始前进行 |
| **代码评审** | 代码提交前（PR Review）或编码过程中进行 |
| **测试评审** | 测试用例编写完成后、执行测试前进行 |
| **上线前检查** | 测试完成后、上线前 24-48 小时 |
| **运维监控** | 上线后持续进行，建议每月/每季度生成度量报告 |

### 6.2 评审粒度

| 场景 | 建议粒度 | 预计耗时 |
|------|---------|---------|
| 需求评审 | 单个用户故事或功能模块 | 30-60 分钟 |
| 设计评审 | 单个微服务或核心模块 | 60-120 分钟 |
| 代码评审 | 单次 PR（<400 行） | 30-60 分钟 |
| 测试评审 | 单个功能模块的用例集 | 30-45 分钟 |

### 6.3 质量纪律

1. **必须先明确质量保障类型**（预防/评审/度量/改进），再路由到对应工作流，不可混用流程
2. **必须覆盖六大方法论**，按阶段侧重应用
3. **每个发现必须标注严重程度**（P0-P3）和优先级
4. **不能只提问题不给建议**，每个风险至少附带一条改进措施
5. **必须区分"设计意图"和"实现风险"**，避免混淆
6. **输出必须使用对应阶段的报告模板**，不可通用化
7. **评审后必须跟踪待办事项的完成情况**，不能评审完就结束
8. **度量必须基于可量化的数据**，不能凭感觉评估质量
9. **改进方案必须有明确的度量指标和验收标准**

### 6.4 知识沉淀

每次评审后更新知识库：

- [ ] 将新发现的缺陷模式补充到 `checklists/review-checklist.md`
- [ ] 将典型案例补充到 `examples.md`
- [ ] 更新 `templates/test-case-template.md` 中的测试策略
- [ ] 更新 `docs/quality-gate-workflow.md` 中的质量基线和检查标准
- [ ] 记录知识更新记录和改进方案

## 7. 常见问题

### Q: 是否所有生成内容都必须经过评审？

A: 建议所有需求、设计、代码在进入下一环节前都经过评审。对于紧急修复或微小改动，可适当简化流程，但仍需进行最小化的检查。

### Q: 评审发现问题太多怎么办？

A: 如果问题数量超过阈值（如 P0/P1 问题 > 5 个），应暂停当前流程，优先修复严重问题后重新评审，避免问题累积。

### Q: 如何平衡评审质量与效率？

A: 建议：
- 使用检查清单（Checklist）确保覆盖度
- 关注高风险的逆向操作、依赖踏空、状态迁移场景
- 对于低风险代码，可适当降低评审深度
