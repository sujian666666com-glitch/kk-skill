# Skill 质量检查清单

## Frontmatter 检查

- [ ] `name` 字段与目录名一致
- [ ] `description` 包含触发关键词（帮助 AI 判断调用时机）
- [ ] `description` 说明了 WHAT（做什么）和 WHEN（什么时候用）
- [ ] `description_zh` 已填写
- [ ] `argument-hint` 简短说明了输入格式
- [ ] `user-invocable` 设置正确（知识库类设为 false）

## 正文结构检查

- [ ] 开头有角色定义（"你是一位..."）
- [ ] 有输入识别逻辑（不同输入走不同路径）
- [ ] 处理流程有明确的步骤编号
- [ ] 每个步骤有具体可执行的指令（非模糊描述）
- [ ] 有输出格式定义
- [ ] 有条件分支处理不同场景
- [ ] 有失败/异常场景的处理方案

## 指令质量检查

- [ ] 避免模糊指令（"请仔细处理"、"确保高质量"）
- [ ] 使用具体量化（"输出必须包含 5 个部分"而非"输出要完整"）
- [ ] 有优先级定义（规则冲突时以哪个为准）
- [ ] 写的是 AI 不知道的领域知识（而非通用常识）
- [ ] 指令可验证（能检查是否符合）

## 输出规范检查

- [ ] 输出格式明确（Markdown/JSON/纯文本）
- [ ] 输出结构完整（定义了所有必需部分）
- [ ] 有输出示例（至少 1 个好的输出范例）
- [ ] 文件保存路径/命名规则明确

## 参考资料检查

- [ ] SKILL.md 控制在 500 行以内
- [ ] 大篇幅模板放在 references/
- [ ] SKILL.md 通过链接引用 references/
- [ ] references/ 文件名有意义（非 readme.md、doc.md）
- [ ] 引用的文件确实存在（链接有效）

## 连接器检查

- [ ] 有 "If Connectors Available" 部分
- [ ] 使用品类名称（文档平台），非具体工具名（Notion）
- [ ] 定义了无连接器时的默认行为
- [ ] 连接器增强描述的是用户价值，非技术机制

## 整体检查

- [ ] 没有拼写错误
- [ ] 没有冗余内容（每句话都有信息量）
- [ ] 逻辑流程清晰（没有跳跃或遗漏）
- [ ] 边界场景有处理方案
